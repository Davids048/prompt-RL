## Baseline Experiment Design

The baseline question is not whether prompt rewriting can be trained well. It is first whether a frozen prompt rewriter changes downstream image or video scores under a fixed generator and fixed evaluator. The target comparison is paired: the same original prompt is generated once directly and once after rewrite, then score deltas are aggregated per prompt before averaging across a small prompt suite. PromptRL is the image-side calibration point because its Table 1 shows Qwen2.5-VL prompt enhancement improves pretrained SD3 but hurts RL-tuned DiffusionNFT and FlowGRPO on several metrics.

```text
prompt suite
    |
    v
rewrite runner: none / Qwen PE / prompt optimizer / video prompt booster
    |
    v
prompt ledger: original prompt, rewrite instruction, rewritten prompt, rewriter params
    |
    v
generator runner: image pipeline or FastVideo T2V pipeline
    |
    v
artifacts: image/video files plus generation metadata
    |
    v
evaluator: image metrics or video metrics
    |
    v
paired score deltas and failure analysis
```

This should be treated as a targeted landscape, not a full factorial grid. The later execution should pick representative cells from the matrix below so the first run is interpretable and affordable.

Current focus questions: 
    First, does prompt rewriting affect video generation models in the same direction and with the same failure modes observed for image generation models? 
        PromptRL shows that prompt enhancement can help a pretrained image generator while hurting RL-tuned image generators, 
        so the immediate T2V question is whether a similar pattern appears under video metrics such as VBench and VideoAlign. 
    Second, which kind of prompt-writer model should be used for the next stage of development? 
        This includes whether the prompt writer should be small or large, text-only or vision-language, 
        generic instruction-tuned or video-specialized, and whether Qwen2.5-VL-style prompt enhancement is 
        enough compared with video prompt boosters such as HunyuanVideo-PromptRewrite, VPO, or Prompt-A-Video.

### Execution Shape

The rewrite engine should be a separate component from generation. It can use Hugging Face Transformers, vLLM, or a hosted API, but it should emit a plain rewritten prompt and a structured log record. For Qwen2.5-VL-3B-Instruct and Qwen2.5-3B-Instruct, the important part is to pin the checkpoint, chat template, system prompt, sampling parameters, max token budget, and output parser. For Promptist, BeautifulPrompt, VPO, Prompt-A-Video, or HunyuanVideo-PromptRewrite, the same wrapper can call the checkpoint-specific generation API and normalize the output to one rewritten prompt string.

For video generation, FastVideo should receive only the final prompt string as `request.prompt` or the CLI `--request.prompt` override. The runner should save the original prompt, rewritten prompt, rewriter model, rewrite instruction ID, generator model ID, seed, frame count, fps, resolution, guidance scale, inference steps, output path, and any failure state. FastVideo's practical starter models are `Wan-AI/Wan2.1-T2V-1.3B-Diffusers`, `Wan-AI/Wan2.1-T2V-14B-Diffusers`, `hunyuanvideo-community/HunyuanVideo`, and FastVideo-provided accelerated variants. SkyReels and VideoCrafter2 are useful literature references, but they should not be first-run dependencies unless a compatible FastVideo pipeline is confirmed.

The evaluator should consume generated artifacts after generation, not during prompt rewriting. That keeps the first baseline as vanilla rewrite rather than search or prompt optimization. A later experiment can add best-of-n rewrite selection, evaluator-in-the-loop rewrite, or RL, but those should be named separately because they change the objective.

### Details To Freeze Or Flag

| Detail / knob                         | Why it matters                                                                                                                                                                  |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Rewrite instruction                   | A "make this cinematic" instruction measures a different thing than "preserve all semantics and only clarify visual detail." The exact system/user prompt should be versioned.  |
| Base vs instruct model                | Base LMs complete text, while instruct/chat models follow role-formatted instructions. The chat template and role placement are part of the condition.                          |
| Rewriter sampling                     | Temperature, top-p/top-k, beams, stop strings, max tokens, and repetition penalty can change prompt length and style enough to move scores.                                     |
| Number of rewrites                    | One rewrite is vanilla rewriting. Multiple rewrites plus evaluator selection is search or best-of-n optimization and should be a separate condition.                            |
| Prompt length budget                  | Long rewritten prompts can be truncated by CLIP/T5 text encoders or dilute important entities and actions. Record tokenized length under the generator tokenizer when possible. |
| Negative prompts                      | Keep them absent or fixed unless the experiment explicitly studies negative prompt rewriting.                                                                                   |
| Generator hidden rewriting            | Hosted generators may silently revise prompts. For this baseline, use generators where the actual prompt is logged or disable hidden rewriting when possible.                   |
| Generator settings                    | Seed, sampler/scheduler, steps, CFG/guidance, resolution, frame count, fps, duration, and model revision must be fixed within each paired comparison.                           |
| Safety filters and failed generations | Rewrites can introduce terms that trigger filters or model failures. Count blocked/failed outputs separately instead of silently retrying.                                      |
| Evaluator leakage                     | If the rewrite instruction includes evaluator rubrics, gains may be judge gaming. The first baseline should keep evaluator details out of the rewriter prompt.                  |
| Score aggregation                     | Aggregate original-vs-rewrite deltas per prompt first, then average. Report whether missing outputs are zero, skipped, or retried.                                              |
| Prompt paraphrases                    | PromptRL's failure mode is prompt overfitting, so include a small paraphrased prompt subset after the first sanity run.                                                         |

GenEval should be run through the cloned reward-server path rather than the shared FastVideo `.venv`, because both official GenEval and reward-server depend on `mmdet` / `mmcv` while FastVideo currently needs Python 3.12 and Torch CUDA 12.9. The remaining blocker is execution access to the existing GPU allocation: `srun --overlap --jobid=4745 nvidia-smi -L` currently fails with `Unable to contact slurm controller`.

For video, these image metrics can be applied to keyframes as diagnostics, but the main video score should still come from native video evaluators such as VBench and VideoAlign because keyframe metrics will miss motion quality, temporal consistency, and text-action alignment failures.
